Visión general

Propósito

lynkko-platform es una librería de infraestructura compartida para el ecosistema Lynkko de SaaS B2B para LATAM. No es una aplicación desplegable — es la capa de paquetes que todas las apps consumen.

Lynkko Ecosystem OS v0.4 posiciona a Lynkko como la capa de operación para el éxito comercial en LATAM: prospección, ejecución de ventas, incentivos internos, comunicación con clientes, membresías y fidelización, centros de ayuda, y cumplimiento de facturación electrónica.

"Lynkko ayuda a negocios en LATAM a vender más con menos desorden."

El ecosistema opera en dos mundos comerciales: Enterprise (Lynkko App, Incentivos, Customer, ClubPass, Help, Facturación) y Pequeños negocios (Turnflow, ClubPass, Facturación, Help). Cada producto puede venderse de forma independiente, pero los productos conectados incrementan el valor total.

Dependencias entre paquetes

Existe exactamente una dependencia inter-paquete: @lynkko/ui → @lynkko/i18n.
Todos los demás paquetes son completamente independientes entre sí.

@lynkko/ui usa getInitials() de @lynkko/i18n para el componente UserAvatar. AppTheme se define de forma idéntica en @lynkko/ui y @lynkko/platform — esta duplicación intencional evita que ui dependa de platform (que tiene drizzle-orm como peer dep).

turbo.json — orden de build

{
  "tasks": {
    "build": {
      "outputs": ["dist/**"],
      "dependsOn": ["^build"]  // ← builds dependencias primero
    }
  }
}

Turborepo detecta que @lynkko/ui depende de @lynkko/i18n y construye i18n primero. Los demás 18 paquetes se buildean en paralelo.

Registry — GitHub Packages

Los paquetes @lynkko/* se publican en GitHub Packages (npm.pkg.github.com). Los paquetes públicos de terceros (tailwindcss, react, drizzle-orm, etc.) siguen viniendo de npmjs.org — esto es estándar de la industria y npmjs.org tiene uptime del 99.99%.

Causa de los errores ERR_INVALID_THIS

El error venía de dos fuentes combinadas:

1. El .npmrc de la app consumidora no tenía @lynkko:registry configurado → pnpm buscaba los paquetes en npmjs.org (donde no existen).

2. pnpm no expande variables de entorno en .npmrc de proyecto. Si el token tenía ${GITHUB_TOKEN} sin expandir, pnpm mandaba un header de auth malformado que corrompía el cliente HTTP para TODOS los registries en cascada, incluyendo npmjs.org.

Solución: token nunca en .npmrc del repo

El .npmrc commiteado solo declara dónde están los paquetes:

# .npmrc  (commiteado en lynkko-platform Y en cada app consumidora)
@lynkko:registry=https://npm.pkg.github.com

El token se inyecta en el entorno de ejecución:

EntornoCómo inyectar el token
Local (~/.npmrc)//npm.pkg.github.com/:_authToken=ghp_TU_PAT
Vercel — Install Commandecho "//npm.pkg.github.com/:_authToken=$GITHUB_TOKEN" >> ~/.npmrc && pnpm install --frozen-lockfile
GitHub ActionsStep: echo "//npm.pkg.github.com/:_authToken=${{ secrets.GITHUB_TOKEN }}" >> ~/.npmrc

PAT requerido

GitHub → Settings → Developer settings → Personal access tokens (Classic):

DestinoScope
Leer paquetes (devs + Vercel)read:packages
Publicar paquetes (CI)write:packages (automático en GitHub Actions de la org)

Pipeline de build y publicación

Build local

# Todos los paquetes en orden
pnpm turbo build

# Un paquete específico
pnpm --filter @lynkko/ui build

# Watch mode
pnpm --filter @lynkko/ui dev

Cada paquete genera tres artifacts via tsup: dist/index.js (CJS), dist/index.mjs (ESM), dist/index.d.ts (TypeScript declarations).

publishConfig en cada package.json

{
  "publishConfig": {
    "registry": "https://npm.pkg.github.com",
    "access": "public"
  }
}

GitHub Actions — .github/workflows/publish.yml

name: Publish packages
on:
  push:
    branches: [main]
    paths: ['packages/**']

jobs:
  publish:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write

    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
        with: { version: 9 }
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: pnpm }

      - name: Inject auth token
        run: echo "//npm.pkg.github.com/:_authToken=${{ secrets.GITHUB_TOKEN }}" >> ~/.npmrc

      - run: pnpm install --frozen-lockfile
      - run: pnpm turbo build

      - name: Publish changed packages
        run: |
          for pkg in packages/*/; do
            cd "$pkg"
            pnpm publish --no-git-checks 2>/dev/null || true
            cd ../..
          done

Patrones transversales

Multi-tenancy

Regla absoluta: todo query que lea o escriba datos de negocio DEBE incluir eq(table.tenantId, ctx.tenantId). No se usa RLS de PostgreSQL — el filtrado es siempre explícito en código.
// ✅ Correcto
const leads = await db.select().from(leadsTable)
  .where(and(
    eq(leadsTable.tenantId, ctx.tenantId),  // ← NUNCA omitir
    eq(leadsTable.status, 'active'),
  ))

// ❌ Incorrecto — leak cross-tenant
const allLeads = await db.select().from(leadsTable)

Audit trail

const audit = createAuditLogger(db)

await audit.log({
  tenantId: ctx.tenantId,
  userId:   ctx.userId,
  action:   AUDIT_ACTIONS.LEAD_CREATED,
  resource: 'lead',
  resourceId: lead.id,
  meta:     { name: lead.name },
  ip:       req.headers.get('x-forwarded-for') ?? undefined,
})

Offline-first

const { isOnline, enqueue } = useOfflineSync()

async function handleCreate(data) {
  if (isOnline) {
    await fetch('/api/leads', { method: 'POST', body: JSON.stringify(data) })
  } else {
    enqueue({ operation: 'create', resource: 'lead',
              resourceId: crypto.randomUUID(), payload: data })
  }
}

  resource_id text,

)

Modelo de datos PEC

Lynkko usa su propio PEC como command center del equipo comercial. Los campos y el pipeline definidos aquí tienen implicaciones directas en el schema de @lynkko/db. Fuente: OS v0.4, §6.

Schema SQL — tabla leads

CREATE TABLE leads (
  id                   uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id            uuid NOT NULL REFERENCES tenants(id),

  -- Identificación
  lead_name            text NOT NULL,
  business_name        text,
  contact_role         text,
  country              text,
  city                 text,
  industry             text,
  business_size        text,        -- '1-10' | '11-50' | '51-200' | '200+'

  -- Origen y clasificación
  source               text,        -- 'meta_ads' | 'linkedin' | 'referral' | 'organic'
  product_fit          text,        -- 'pec' | 'turnflow' | 'clubpass' | 'incentivos' | ...
  product_interest     text[],      -- múltiples productos de interés
  pain_signal          text,        -- frase literal del prospecto
  channel              text,        -- 'whatsapp' | 'email' | 'call' | 'linkedin'
  campaign             text,

  -- Scores (calculados por el AI agent)
  fit_score            smallint CHECK (fit_score    BETWEEN 0 AND 100),
  intent_score         smallint CHECK (intent_score BETWEEN 0 AND 100),
  icp_tag              text,        -- ver tabla ICP_TAGS

  -- Estado del pipeline
  current_stage        text NOT NULL DEFAULT 'sourced',
  owner                text,
  last_interaction_at  timestamptz,
  next_action          text,
  next_action_at       timestamptz,

  -- Conversación
  objections           text[],
  notes                text,

  created_at           timestamptz NOT NULL DEFAULT now(),
  updated_at           timestamptz NOT NULL DEFAULT now()
);
